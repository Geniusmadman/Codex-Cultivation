## 结论

- 已实现 `verify`、`install`、`set-realm`、`remove` 四个管理命令。

## 未完成项

### P0：生成四套正式境界图集

筑基、金丹、元婴、化神需要分别生成独立的完整 Pet v2 图集，不能继续复用炼气文件。

每个境界都必须完成：

- 9 行标准动画和 16 个顺时针观察方向。
- `1536 x 2288`、`8 x 11`、`192 x 208` 单元格契约。
- 同一张脸、耳形、主体毛色、额心月纹和整体风格的家族一致性。
- 对应境界的尾数、毛纹、胸前月印和额纹差异。
- 独立的 `spritesheet.webp` 和 `validation.json`。
- 动画预览、方向语义、连续性、透明度、色键残留和最终接触表检查。
- `hatch-pet` 要求的确定性校验、方向盲测和最终视觉 QA。

完成后必须删除四个 `placeholderSourceRealm` 标记，并确认五个图集不再具有相同 SHA-256。

### P0：完成五境 Windows 实机验收

现有测试证明代码和文件状态链路可以工作，但还没有用五套真实外观完成以下实机闭环：

- 银月宠物窗口未打开时切换境界。
- 银月宠物窗口已经打开时自动热重载。
- 从炼气依次切换到筑基、金丹、元婴、化神，并确认屏幕上的外观真实变化。
- 热重载失败时保留 `pendingReload`，托盘正确提示重启。
- 用户确认重启后加载目标境界资源并清除 `pendingReload`。
- 完整恢复默认移除银月，`-KeepSpiritPet` 保留银月。
- 全程确认其他内置宠物和用户宠物的文件哈希不变。

### P1：校准文档和发布口径

README 和 Changelog 已描述“五境自动进化”，但当前只有五境代码映射，四境美术仍为占位。正式发布前需要统一改为真实完成状态，避免把“境界切换逻辑已通过”写成“五境美术已完成”。

正式发布说明至少需要明确：

- 五个境界均使用不同的正式资源。
- Windows 实机五境切换已经验收。
- macOS 是否支持灵宠，或明确标记为不支持。
- 用户需要在 Codex 原生宠物选择器中选择一次“银月”；安装器不会强制替用户切换当前宠物。

### P1：完成 Git 交付

当前灵宠实现和占位资源仍是本地工作区改动，没有形成最终功能提交，也没有推送。现阶段符合“本地验证、不上传”的要求，但不属于可交付完成状态。

### P2：macOS 灵宠支持

当前银月管理、安装、同步、托盘和恢复链路只实现了 Windows。macOS 修仙界面已经存在，但银月灵宠不在当前 macOS 实现范围内。若未来要求跨平台，需要单独设计 macOS 宠物目录、启动生命周期、窗口识别、状态路径和恢复流程。

## 使用边界

- 只使用原生银月：在 Codex 宠物选择器中选择“银月”即可显示当前已安装形态。
- 使用界面与境界联动：必须通过 `Codex Cultivation` 快捷方式或 `start-codex-cultivation.ps1` 启动。
- 当前切换到筑基、金丹、元婴、化神时，状态和资源文件名会变化，但外观仍与炼气相同。
- 安装器只安装和管理银月，不会自动替用户选择或打开宠物窗口。

- 五个境界各自拥有独立、通过校验的 Pet v2 图集。
- 四个占位标记全部移除，五个图集不再复用同一文件。
- 五境自动同步、热重载、失败待重启和恢复流程完成 Windows 实机验收。
- 其他宠物在安装、进阶和恢复前后的文件保持不变。
- README、Changelog 和实际能力一致。
- 完整自动回归通过，最终功能改动形成可审查的 Git 提交。
